Project Overview

The aim of this project is to analyze a database containing recipes from around the world using networks. With this we intentend to find similarities between the database and the real world.

Data visualization through ingredient and recipe networks.

The Database

The database for this project was obtained from a Kaggle competition and Yummly and it contains recipes from all over the world. In total we have 39774 recipes and 1817 ingredients. We will be working with a JSON file of 12Mb. All the data inside will be ordered by cuisine, recipe ID, and ingredients of the recipe. You can download the database HERE.

Creating the network

To start creating the network, first we needed to define which were going to be the nodes and which would be the links between them. In our case, we decided that the ingredients of each recipe would be the nodes of the network. The links on the other hand would be the recipes they share. This means that, if two ingredients appear in the same recipe they would be linked in the network. Moreover, we decided to create the network as a weighted network, where the weight of the links would be the number of recipes the ingredients share. In total, we had 39774 recipes and 1817 ingredients. With this, we created a network with 1817 nodes and 41325 links. As mentioned before it is a weighted and undirected graph.

The next step was to assign each ingredient to a cuisine. Up to this point, what we have in the network is a mess of ingredients mixed up with links connecting them. Since our goals are to compare this network to reality we need to assign each ingredient to a different cuisine.

From our data, we can extract that we have 20 different types of cuisines: irish, mexican, chinese, filipino, vietnamese, moroccan, brazilian, japanese, british, greek, indian, jamaican, french, spanish, russian, cajun creole, thai, southern US, korean and italian. To assign each ingredient to its cuisine we need to check in which recipes the ingredient appears and add it to the cuisine that the recipe belongs to. However, many ingredients aren't unique to a cuisine. To solve this issue we aproached the solution in two ways and after seeing the results we selected the one we thought was most representative of reality. Here we will explain the method we finally used, but the other one can be found in the notebook.

The method we chose assigned unique ingredients (ingredients that only appear in recipes from one cuisine) to their respective cuisine, and ingredients that appear in multiple cuisines to the one in which they appeared most. Let's see this with an example from the notebook:

Let's dive deeper into the meaning of the computed indicators. First, what's the difference between degree and degree centrality?

Degree centrality of a node n is degree(n)/numberof_nodes, where degree(n) is the _simple degree -ignoring the weights! Let's take as an example two nodes that have the same degree. Degree centrality is giving more importance to the ingredient that is connected to a higher number different ingredients, over an ingredient that shares many recipes with a fewer number of ingredients. That explains the differences between these two rankings.

Betweenness centrality: as we know betweenness centrality quantifies the number of times a node acts as a bridge along the shortest path between two other nodes. In our network, it quantifies the number of times an ingredient is on the way between two ingredients, if we tried to navigate between ingredients following the edges denoted by the shared recipes. Even if it measures the centrality of a node, it doesn't have a clear physical meaning in our dataset.

Eigenvector centrality assigns relative scores to all nodes in the network based on the concept that connections to high-scoring nodes contribute more to the score of the node in question than equal connections to low-scoring nodes. That is, being connected to a hub or a node that is close to a hub gives a high score. Again, it's not a bad measure of centrality but it doesn't carry a strong stand-alone meaning.

Examining the generated text file, we see that there are subtle differences in the top ten of values for the different indicators. The difference though, is mainly the order of the ingredients, but in each cuisine we see that the ingredients in the different top 10s are almost the same, so we'll consider them all good to get us to our goal. These rankings let us, as we hoped, see the most important ingredients of each cuisine!

Communities

Our next goal is to analyse the communities present in our network. Our first hypotheses is that the cuisines are communities that show the modularity of the network. We will start by computing the modularity of the cuisine partition. We'll then find the best partition with Louvain's method and then compare the obtained modularity -which is the optimal since the best partition is the one that maximises the modularity.

The result from doing this are the following ones:

The modularity for the louvain partition is 0.285239.

The modularity for the cuisine partition is 0.209728.

The modularity for a partition found randomly is 0.069611.

We see that the Louvain partition achieves a 36% higher modularity than the cuisines, so we found a slightly better way to group the ingredients than the cuisine they belong to. According to Louvain's heuristics, of course. The modularity of the cuisine partition isn't low enough to say it's a bad partition, we see that it indeed shows part of the community structure present in the network. It is, certainly, much higher than the modularity of a random partition, which doesn't come as a surprise.

From the confussion matrix we generated, that we are not going to link here because of the size (you can find it in the notebook), we can observe a few interesting things.

The Louvain partition contains 15 communities. That is 5 less communities than cuisine types we have, so cuisine types have been merged to form broader communities. As a perfect example let's take a look at the 4th Louvain community (4th row of the confusion matrix). It has a big ammount of chinese ingredients, followed by many japanese, thai, vietnamese and korean ingredients. In fact, we see that the asian cuisine is one single community according to Louvain's partition! The nodes belonging to asian types of cuisine aren't separated enough between each other too be considered different communities according to Lovuain's heuristics.

Similar effects can be seen in other rows of the confusion matrix, like the 1st one, that contains a high number of italian ingredients but also french and mexican, suggesting that there are many common ingredients between recipes of those cuisines.

Solving the recipe name problem

At the beggining of the project we have mentioned that the database didn't include the names for the recipes, only ID's. The dataset we've been using has allowed us so far to find the most common ingredients, the most representative ingredients of the different cuisines, to check that cuisines form communities, and to find a partition better than the cuisines. What else could we achieve with it? One could think of comparing recipe names across cuisines, or to build a reccommender of a recipe given certain ingredients -or another recipe. But for some of these applications we would do better if the dataset included the recipe names.

That gave us the idea of trying to generate a name ourselves. We'll build a function that comes up with a name for the given recipe. We don't expect great results, but it is a good way of applying the measures and indicators we've computed so far and to see how useful they are in overcoming the expected inconvenients of our limited method. Note that we don't have the recipe preparation steps either so we can't really know how are the ingredients combined. We'll also add a touch of humor to the generated names. They will consist basically of two important ingredients of the given recipe, joined by some preposition and embellished by -often pompous- adjectives.

Sentiment Analysis

Analysis of the sentiment realted to each cuisine on Twitter.

Introduction to sentiment analysis

In this section we are going to make some sentiment analysis of the different cuisines with Twitter. We have gathered more than 1000 tweets from each cuisine, searching with the hashtag "#spanishcuisine" for example for spanish cuisine using the Twitter API. After cleaning all the tweets, we took the text from each one and analyzed the sentiment based on the LabMt wordlist, that contains an average sentiment for a number of words. This are the results for the analysis:

As we can see from the graph on the left, all the cuisines are quite close in the range of 5 to 6 average sentiment. However, if we adjust the range of the axis like the graph on the right, we can see the differences a bit more clearly. The fist thing to notice is that the sentiment related to american a british cuisines is significantly lower than the rest. In a second level, with an average sentiment level we find mexican, italian, greek or japanese cuisines , and above those with the best sentiment are chinese, cajun or vietnamese cuisines. We find quite interesting that cuisines with usually "bad reputation" like the british and american ones are the ones with the lowest rating. On the ohter hand asian quisines have good reputation in the western countries, so that might be a reason why they are rated so high.

Let's now take a look at this in the world map.
